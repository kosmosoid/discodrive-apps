# DiscoDrive sync daemon and E2E vault

Cross-platform DiscoDrive sync client (Dropbox model: folder ↔ server) with a client-side E2E vault.

## Daemon

```
discodrive pair --server https://files.example.com --dir ~/discodrive   # pair via browser
discodrive run        # background bidirectional sync (push→pull, fsnotify+SSE)
discodrive run --detach   # same, detached from console (log written to config dir)
discodrive status     # synced / in progress / offline
discodrive install    # autostart (launchd on macOS / systemd --user on Linux)
discodrive uninstall
```

Build with menu-bar/tray icon (requires CGO, behind a build tag):
```
go build -tags tray -o discodrive ./cmd/discodrive
discodrive tray       # icon: status, open folder, vaults (create/open/close)
```
Without the tag, builds a pure-Go binary (headless, cross-compile without cgo).

## E2E vault — Cryptomator-compatible

The vault is encrypted **on the client**: the server stores only ciphertext and encrypted names. The format is
**Cryptomator vault format 8 (cipherCombo SIV_GCM)**, byte-for-byte compatible. This means:

- **Freedom and no lock-in.** The same vault can be opened:
  - natively — in DiscoDrive (tray daemon and desktop client: create/open/close);
  - with **official Cryptomator apps** (desktop, iOS, Android) — just point them at the vault folder as an existing vault, even when the server is unavailable (anti-lock-in by design).
- **Recovery key** — standard Cryptomator phrase (44 words): a key generated by our code is
  accepted by Cryptomator, and vice versa.

### Security guarantee (tray vault)
An open vault is decrypted into the system cache (`<UserCacheDir>/discodrive/open/`) — **outside the sync folder**,
so the daemon physically cannot see the plaintext and cannot upload it to the server. Closing re-encrypts
any changes back and wipes the plaintext.

### Web vault (read-only)
Decryption happens **in the browser** (WebCrypto); the password never leaves the client. Caveat: web E2E is weaker
than native — the page code is served by the server, so a server compromise or XSS could theoretically intercept
the password at entry time. For maximum security, use the native client. The web vault is a convenience viewer.

## Structure
- `internal/index` — local state index (SQLite, modernc — pure Go).
- `internal/engine` — sync core (pull/push/conflicts, "server is truth").
- `internal/protocol` — HTTP client to the server (device token → JWT, SSE).
- `internal/syncer` — background loop (triggers, backoff, status).
- `internal/vault` — vault crypto format (Cryptomator fmt8) + recovery key.
- `internal/vaultmgr` — vault lifecycle in the daemon (open/close, plaintext outside SyncDir).
- `internal/desktop` — on-demand controller for the cross-platform Fyne desktop client.
- `cmd/discodrive` — daemon/CLI/tray. `cmd/discodrive-desktop` — Fyne on-demand GUI.
